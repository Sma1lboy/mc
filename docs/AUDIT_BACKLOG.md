# Launcher chain audit backlog

Generated 2026-06-22 from a multi-agent end-to-end audit (10 chains, adversarially verified).
38 confirmed gaps. Ranked by user-value / effort; correctness bugs kept high.

## Shipped 2026-06-22

- **#1** Workspace-layout account login — `AccountDialog` (theme-neutral) reachable from ContextBar + Rail.
- **#2** Export writes the clean loader build (`47.2.0`), not the dir-id, so Forge/NeoForge mrpacks re-import.
- **#3** Installing a mod version removes the superseded jar (no duplicate-modid crash). `mods::remove_superseded`.
- **#4 + #5** Process registry + `game://started`/`game://exit` events: real running state, Stop, crash diagnostics, no success-on-first-log, double-spawn guard.
- **#6** Auto-provision a matching Temurin JRE on launch when none is installed (`LaunchSpec.runtimes_dir`).
- **#7** World backup uses save-as (no silent overwrite). **#23** rename re-entrancy guard.
- **#15** Incompatible mod versions can't be installed (button blocked + backend-agnostic guard).

Remaining: see ranks #8–#14, #16–#22, #24–#31 below.

## Cross-cutting themes

- No exit/crash/process lifecycle anywhere: the spawned child is detached and never tracked (commands.rs:706-708), so 'running' is permanently false, the Stop UI is dead, exit codes are discarded, no game://exit event exists, and the success toast fires on the first log line — fix all of these with one process-registry + exit-event sweep (ranks 4, 5).
- Dead/unreachable backend wired to no UI — a recurring pattern across chains: remove_account, install_jre auto-provisioning, YggdrasilClient, the conflict-aware resolver, custom_roots, and InstanceManageDialog (in the workspace layout) are all fully built in mc-core/commands but called from zero or one UI surface. A single 'audit every #[tauri::command]/api.ts method for a reachable caller in BOTH layouts' pass would surface them.
- Workspace vs classic layout parity is systematically broken: account creation/switch/remove, per-instance management (settings/mods/packs/worlds/copy), instance delete, and datapack browse all exist in exactly one layout. Treat the two layouts as a contract — every instance/account action must be reachable in both — and sweep InstanceRow/ContextBar/Home/Library against ClassicLaunch.
- Silent success on broken outcomes: a green/success toast fires even when the operation produced something non-functional — wrong-loader/wrong-version mod install (rank 15), per-version install missing deps (rank 16), datapack to an unread dir (rank 8), blocked CurseForge files (rank 18), and modpack import that 404'd mid-way (rank 11). Add a shared 'did this actually work?' gate before toasting success across the install/import chains.
- Global Settings controls that are persisted but never read: java_path, default_memory_mb, and custom_roots all round-trip through get/set_settings yet only the downloader reads GlobalSettings (concurrency/mirror). One sweep to thread GlobalSettings into create_instance, resolve_java, and discover_roots closes ranks 12 and 29, and fixes the 'blank = follow global' label that currently lies.
- Token lifecycle is unfinished end-to-end: expiry (expires_in) is captured then discarded, MsaClient::refresh has zero callers, YggdrasilSession.client_token has nowhere to persist, and tokens sit in plaintext — refresh/validate are unreachable for every account kind. A single auth-persistence pass (add expires_at + client_token, wire refresh, move secrets to a keyring) addresses ranks 9, 28, 30.
- Error handling swallowed or shown as a bare string: screenshot loadUrl has an empty catch (perpetual spinner, rank 24), launch failures surface as `启动失败:${e}` with no recovery, and JavaNotFound returns a misleading 'auto-install is disabled' message. Standardize an error surface (placeholder + retry, actionable recovery) instead of empty catches and raw interpolated errors.
- List/result truncation with no way to reach the rest: version lists capped at 24 (rank 21) and search capped at 30 with no pagination (rank 22), even though the backends return/​support more (and ModpackDetail already virtualizes). One virtualization + offset-pagination sweep across Discover, ProjectInstallDetail, and the in-instance search surfaces.

### 2. Export writes the instance dir id as the loader version, corrupting Forge/NeoForge mrpacks on re-import  `[high/medium]` (modpack)
- **Why:** Silent data-corruption correctness bug: every exported Forge/NeoForge pack is un-reinstallable here and in other launchers; bugs near top regardless of effort.
- **Evidence:** instance/mod.rs:353-356 loader_version = Some(dir_id) where dir_id is the slugified display NAME; ClassicLaunch.tsx:177 & util/instanceActions.ts:65 pass it verbatim; export_modpack (commands.rs:803-847) trusts it; export/modrinth.rs:148-164 build_dependencies writes `"forge":"<slug>"`; on re-import import/modrinth.rs:120-136 reads it as build, loader/core.rs:50-55 feeds install_forge/install_neoforge → bogus installer URL fails. Fabric/Quilt survive only because core.rs:44-49 ignores their pinned version.
- **Fix:** Resolve the real loader build from the instance's version json inheritsFrom chain (core_id like '1.20.1-forge-47.2.0') in export_modpack instead of trusting InstanceSummary.loader_version; OR stop overloading loader_version with dir_id in instance/mod.rs (resolve the true build from the version profile). At minimum, refuse to emit a forge/neoforge dependency value that isn't a real build number.

### 3. Installing a different version of a mod leaves the old jar → duplicate-modid launch failure  `[high/medium]` (mods)
- **Why:** Common correctness bug (update Sodium 0.5.3→0.5.8 from the version list) that produces a non-launching instance; high value, contained fix.
- **Evidence:** install_mod_version (crates/mc-core/src/instance/install_mod.rs:86-111) writes file.filename with no cleanup of a prior jar for the same project; install_mod (install_mod.rs:121) and install_version_file (commands.rs:226-237) likewise never remove a prior file. The '已存在' toast (InstanceManageDialog.tsx:761-762) only fires when nothing was installed, so it is not a guard.
- **Fix:** In install_mod_version, before/after writing the new file, locate any existing installed jar for the same project (track project_id→file_name, or match by mod_id from list_mods) and delete the stale one when its filename differs. ### 4. Game exit/crash never reported to UI; success toast fires on any first log line  `[high/medium]` (launch)
- **Why:** The UI actively misreports the most common failure mode (mod/Java/library startup crash) as success; mc-core diagnostics already exists and the CLI sibling already prints exit status — pure wiring for big trust win.
- **Evidence:** commands.rs:706-708 reaps child as `let _ = child.wait().await;` discarding ExitStatus; no game://exit event exists (grep). Home.tsx:66-71 fires '游戏已启动' on the FIRST game://log line regardless of level (stderr lines emitted level:'error' at commands.rs:701 still trigger it). ClassicLaunch.tsx:92 only appends raw lines, and setLaunching(false)/'已启动' fire right after the spawn-only IPC returns. diagnostics.rs analyze_exit(exit_code, log) (line 309) is used only by mc-cli (main.rs:562); cli cmd_launch (main.rs:730-731) already does child.wait() and prints exit status.
- **Fix:** After child.wait() resolves, emit `game://exit` { code, success }. On the UI clear running state; on non-zero, feed the captured stderr through diagnostics::analyze_exit to surface a human-readable crash reason. Gate the success toast on a real running signal, not the first log line.

### 6. Java auto-provisioning built in core but never called on launch; wrong Java major = dead-end error  `[high/medium]` (launch)
