# 03 · 技术栈畅想(性能 / 效率优先)

> 抛开三个参考实现的历史包袱(C++/Qt 重、VB.NET 锁死 Windows),站在"如果今天从零开始、把性能和效率拉满"的角度大胆畅想。

## 0. 先想清楚:启动器的性能瓶颈在哪

启动器**不是** CPU 密集型应用。它的"性能/效率"体现在四个真实瓶颈上,选型要对着这四个打:

| 瓶颈 | 本质 | 对技术栈的要求 |
|------|------|----------------|
| **冷启动速度** | 启动器自己打开要快(用户最敏感) | 小体积、原生、无 runtime 预热(JVM/Electron 是反例) |
| **下载吞吐** | 几百个文件、上 GB,要打满带宽 | 高并发 IO、async 运行时、零拷贝、连接复用 |
| **文件校验** | 上千文件算 SHA1,易卡 UI | 多核并行、SIMD 哈希、不阻塞 UI 线程 |
| **内存占用** | 后台常驻,不该吃几百 MB | 无 GC 抖动、无 Chromium 进程、紧凑数据结构 |

结论:**真正的性能差异在"核心层",不在 UI 层**。所以选型策略是——**核心层用系统级语言压榨性能,UI 层选能快速做出好看界面的方案**。

## 1. 大胆畅想:Rust 核心 + Tauri 外壳(主推)🦀

```
┌─────────────────────────────────────────────┐
│  UI 层:Tauri v2 (系统 WebView) + 前端框架     │  ← 体积 ~10MB,非 Electron
│    SolidJS / Svelte(比 React 更轻更快)        │
├─────────────────────────────────────────────┤
│  IPC:Tauri command(Rust ↔ 前端,零拷贝大数据)│
├─────────────────────────────────────────────┤
│  核心层:纯 Rust crate(无 UI,可独立测试/CLI) │
│    ├─ tokio          async 运行时(高并发下载)│
│    ├─ reqwest/hyper  HTTP + 连接池 + HTTP/2    │
│    ├─ rayon          多核并行(批量 SHA1 校验) │
│    ├─ sha1/sha2 (asm) 硬件加速哈希             │
│    ├─ serde_json     版本 json 解析(极快)     │
│    └─ zip/flate2     natives 解压              │
└─────────────────────────────────────────────┘
```

**为什么是它(对着四个瓶颈)**:

| 瓶颈 | Rust+Tauri 怎么赢 |
|------|-------------------|
| 冷启动 | 原生二进制,无 runtime 预热;Tauri 用系统 WebView,不打包 Chromium → 启动 < 0.5s,体积 ~10MB(Electron 同类 ~150MB) |
| 下载吞吐 | tokio + reqwest 连接池,几百并发轻松;Rust 无 GC,IO 密集场景吞吐稳定 |
| 校验 | rayon 把上千文件 SHA1 摊到所有核心,sha1 crate 走 CPU SHA 扩展指令,几秒搞定;全程在 Rust 线程,UI 不卡 |
| 内存 | 无 GC、紧凑结构,常驻几十 MB |

**额外红利**:
- **核心层天然可复用**:纯 Rust crate 不依赖 UI,能直接编译成 CLI、能跑单元测试、未来能做 headless 服务端(整合包 CI、专用服)。这正是 PCL-CE 想用"核心库分离"达到但受限于 C#/Windows 的目标,Rust 做得更彻底。
- **跨平台一份代码**:Win/macOS/Linux + arm64 全覆盖,native 库和 Java 架构匹配在 Rust 里用 `cfg!` 优雅处理。
- **前端生态做 UI 快**:主题、动画、Mod 列表这些 UI 重活,用 Web 技术(CSS/SolidJS)比 Qt/WPF 迭代快得多,还能复用海量组件库。

**代价(诚实说)**:
- Rust 学习曲线陡,async + 生命周期上手有成本。
- Tauri 的 WebView 跨平台渲染有细微差异(各系统 WebView 内核不同),复杂动画需测。
- Mod 安装、loader installer 偶尔要跑 Java 进程,这部分 Rust 只是"调用方",没难度。

## 2. 备选 A:Rust 核心 + 纯 GPU 原生 UI(极致性能)

UI 也不用 WebView,用 Rust 原生 GPU UI(`egui` / `iced` / `slint`)。

- **优点**:彻底无 WebView,体积更小(~5MB)、启动更快、内存更低、一种语言到底。
- **缺点**:原生 GUI 生态远不如 Web,做"好看且动效丰富"的界面(PCL 那种精致感)成本高;`egui` 偏工具风,`slint` 较新。
- **适合**:追求极简、不在意花哨 UI、想要单语言洁癖的场景。

## 3. 备选 B:.NET 8 + Avalonia(团队熟 C# 时)

如果团队栈是 C#,又想跨平台(摆脱 PCL2 的 VB.NET/WPF/Windows 锁定):

- **核心层**:.NET 8 C#,可直接**移植/参考 PCL-CE 的 `PCL.Core`**(已经是 C#,IdentityModel、源码生成配置、下载框架现成)。
- **UI**:Avalonia(跨平台 XAML)或 MAUI。
- **性能**:.NET 8 的 NativeAOT 能编译成原生二进制,启动快、无 JIT 预热、体积可控,性能接近原生(虽仍不及 Rust,但远好于 Electron/JVM)。
- **效率红利**:**能最大化复用 PCL-CE 的现成代码**,开发速度最快。
- **取舍**:性能上限低于 Rust,GC 仍在(但启动器负载下影响小)。

> 这是"性能尚可 + 复用现成 PCL.Core + 开发最快"的务实之选。

## 4. 三方案对比

| 维度 | ① Rust + Tauri(主推) | ② Rust + 原生 GPU UI | ③ .NET8 + Avalonia |
|------|----------------------|---------------------|--------------------|
| 启动速度 | 极快(<0.5s) | 最快 | 快(AOT) |
| 安装体积 | ~10MB | ~5MB | ~30-60MB |
| 运行内存 | 很低 | 最低 | 中 |
| 下载/校验性能 | 顶级(tokio+rayon) | 顶级 | 良好(并行+async) |
| UI 开发效率/美观 | 高(Web 生态) | 低 | 中高(XAML) |
| 跨平台 | 全 | 全 | 全 |
| 代码复用参考 | 重写,参考 Prism 抽象 | 重写 | **直接复用 PCL.Core** |
| 学习曲线 | 陡 | 陡 | 平(若懂 C#) |
| 推荐场景 | 全新打造、要性能+好看 | 极简洁癖 | 团队熟 C#、要快出活 |

## 5. 我的大胆畅想结论

**首选 ① Rust 核心 + Tauri:** 它精准命中启动器的四个真实瓶颈(冷启动、下载、校验、内存),把性能拉满的同时,用 Web 前端保住 UI 的迭代效率和美观度,跨平台还白送。核心层做成独立 Rust crate,这是三个参考实现都想要、但都没完全做到的"纯净可复用内核"。

**落地建议**:
1. **先把核心 crate 跑通**(无 UI):`mc-core` —— 版本解析 + 下载 + 校验 + Java 检测 + 命令行拼接 + 起进程。用一个 CLI(`mc-core launch <instance>`)验证能启动游戏。这一步不碰 UI,纯逻辑,最容易测、最能压性能。
2. **核心稳了再套 Tauri UI**:UI 通过 Tauri command 调核心,大数据(下载进度流、日志流)用 Tauri 的 event/channel 推送。
3. **性能基线**:从第一天就压"1000 文件校验耗时""下载吞吐 vs 带宽""启动器冷启动 ms"三个数,作为回归指标。

> 若团队没有 Rust 经验、又想尽快出可用版本,退而求其次选 ③(复用 PCL-CE 的 `PCL.Core`),后续核心层可逐步用 Rust 重写热点。
