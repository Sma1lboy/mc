Agent 架构收拢说明

把工具执行收回到客户端，把脑子留在 agent-core。

这次改动的核心不是“再写一个 Modrinth API”，而是把整条 agent 工具链改成 agent-core 只负责模型 loop + 工具定义，真正的执行全部走 launcher client 的 Rust IPC。这样 UI、Claude Code、本地/远端迁移都能复用同一条协议，不会再有两份 truth。

前后对比

我们真正改掉的边界

改之前

  • TS 里有 executor / mock / modrinth 两套工具执行。
  • agent-core 既定义工具，又夹带执行逻辑。
  • 桌面端和本地 runtime 各自维护自己的工具路径。
  • 未来移到远端 server 时，重复搬运成本高。
→

改之后

  • agent-core 只保留 tool schema 和模型 loop。
  • 所有工具都视为 client-side tool，`input-available` 后由客户端执行。
  • 桌面 UI 通过 Rust IPC 回填 tool output。
  • Claude Code 本地 runtime 通过 stdio bridge 复用同一路径。

执行链路

一次 tool call 的生命周期
agent-core

发出工具调用

模型只看到 tool schema，不知道执行细节。turn 在 `input-available` 停住。

→
desktop client

统一 dispatch

chatStore / localRuntimeAdapter 捕获 tool call，按名字走 Rust IPC 或 UI 交互。

→
Rust daemon

执行并返回结果

Modrinth、实例列表、build 等能力都回到 Rust，结果写回消息后再续跑。

这条链路的意义是：以后 agent-core 搬到 server 时，客户端执行侧不需要重做协议，只需换 transport。

这次具体做了什么

按层拆开看
1. agent-core

把工具改成纯定义

  • 删掉 `executors/*`，不再在 TS 里写 Modrinth 实现。
  • 每个 tool 文件只保留 schema / description。
  • `createModpackAgent(settings)` 只负责模型 loop。
  • 本地 Claude Code 仍可通过 bridge handler 代理 tool call。
2. desktop

统一工具分发入口

  • 新增 `clientToolDispatcher.ts` 作为单一分发器。
  • `chatStore` 自动执行自动工具并把结果 append 回消息。
  • `localRuntimeAdapter` 把 Claude Code 的工具请求接回客户端。
  • 交互类工具继续由 UI 组件完成。
3. Rust

保住 truth source

  • 把 agent 能力重新收束到 Rust IPC 命令。
  • 搜索、解析、构建、实例信息都只认 Rust。
  • 未来远端 server 只需要换 transport，不需要重写业务。

结论

一句话总结
这次不是把 agent 再“做一遍”，而是把它变成一个更干净的协议层： agent-core 负责推理和工具定义， desktop 负责把 tool call 落到客户端能力， Rust 负责真实执行。 这样现在是单一 truth，未来迁到远端 server 也只换 transport，不换心智模型。
